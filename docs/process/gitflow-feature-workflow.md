# Gitflow Feature Workflow

Этот файл задает короткий и обязательный процесс разработки новых фич в проекте.
Это практическая инструкция для `feature/*` веток внутри текущего `gitflow`.

## Основной принцип

Каждая новая фича делается не хаотично и не сразу через код, а через последовательность:

`milestone -> scope -> feature branch -> plan -> improve -> implement -> verify -> review -> commit -> merge`

AI Factory используется не как генератор случайных изменений, а как управляемый цикл разработки.

## Обязательные правила

- Каждая новая продуктовая работа начинается только в отдельной ветке `feature/*`.
- Одна ветка должна соответствовать одному логическому feature-flow.
- Перед кодом должен существовать branch-scoped план.
- Бизнес-логика разрабатывается с TDD-подходом.
- Критические пользовательские сценарии обязаны иметь smoke или e2e покрытие.
- Merge в интеграционную ветку допустим только после `verify` и `review`.
- Если scope поменялся по ходу работы, сначала обновляется план, потом продолжается реализация.
- После merge branch-scoped план переносится в `docs/process/archive/ai-plans/`.

## Когда можно начинать новую фичу

Новую фичу можно начинать только если:

- следующий milestone понятен в `ROADMAP.md`;
- есть acceptance criteria или хотя бы ясный scope;
- текущая завершенная работа уже влита;
- новая ветка не смешивает несколько независимых задач.

## Обязательный цикл для каждой фичи

### 1. Выбрать следующий milestone

Проверь:

- `.ai-factory/ROADMAP.md`
- `.ai-factory/RULES.md`
- при необходимости `.ai-factory/DEVELOPMENT-PLAYBOOK.md`

Если milestone неясен или scope спорный:

```text
/aif-explore <тема фичи>
```

### 2. Создать feature-ветку

Создавай отдельную ветку вида:

```text
feature/<short-name>
```

Примеры:

- `feature/learning-content`
- `feature/profile-test`
- `feature/recommendation-flow`
- `feature/analytics`

### 3. Построить branch-scoped план

Для каждой новой ветки сначала создается план:

```text
/aif-plan full --parallel <описание фичи>
```

Пример:

```text
/aif-plan full --parallel Implement feature/learning-content for nps-choice-platform: structured learning content across direction pages and comparison, content model updates, empty states, unit/integration coverage, smoke coverage if user flow changes
```

### 4. Улучшить план перед кодом

Сразу после создания плана:

```text
/aif-improve
```

Цель шага:

- убрать пробелы;
- проверить зависимости;
- уточнить тесты;
- уточнить acceptance criteria;
- убедиться, что фича не ломает архитектурные границы.

### 5. Реализовать по плану

После фиксации плана:

```text
/aif-implement
```

Во время реализации:

- не перепрыгивать через план;
- не смешивать unrelated changes;
- держать фичу в рамках своей ветки;
- при изменении требований сначала обновлять план.

### 6. Проверить результат

После завершения реализации:

```text
/aif-verify
```

Проверка должна подтвердить:

- задачи плана действительно закрыты;
- ничего не забыто;
- тесты, сборка и quality gates проходят;
- нет архитектурного дрейфа относительно проекта.

### 7. Провести review

Перед коммитом и merge:

```text
/aif-review
```

Review должен в первую очередь искать:

- баги;
- поведенческие регрессии;
- нарушения архитектурных границ;
- недостающие тесты;
- рискованные assumptions.

### 8. Зафиксировать изменения

После зеленого verify и review:

```text
/aif-commit
```

Коммит должен:

- соответствовать содержанию ветки;
- быть в формате Conventional Commits;
- не смешивать несколько независимых смыслов в одном сообщении.

## Что делать при проблеме после merge

Если баг найден после интеграции:

```text
/aif-fix <описание проблемы>
```

Использовать fix flow нужно обязательно для:

- регрессий;
- сломанных пользовательских сценариев;
- ошибок интеграции между уже влитыми фичами.

## Когда обновлять контекст проекта

Используй специальные команды только по назначению:

- `/aif-roadmap` - если изменился порядок milestones
- `/aif-rules` - если появилась новая обязательная командная договоренность
- `/aif-docs` - если изменилась пользовательская или инженерная документация
- `/aif-evolve` - после серии повторяющихся ошибок или патчей, чтобы улучшить проектный skill-context

## Чего делать нельзя

- Нельзя начинать кодить крупную фичу без `/aif-plan`.
- Нельзя вести новую продуктовую работу напрямую в общей ветке.
- Нельзя считать milestone автоматически равным одной ветке.
- Нельзя делать merge без `verify`.
- Нельзя закрывать задачу без проверки пользовательского сценария, если она влияет на основной flow.
- Нельзя добавлять новую фичу в release stabilization phase.

## Короткий шаблон на каждую новую фичу

```text
1. Проверить ROADMAP.md
2. Создать feature/<name>
3. /aif-plan full --parallel <описание>
4. /aif-improve
5. /aif-implement
6. /aif-verify
7. /aif-review
8. /aif-commit
9. Merge
```
