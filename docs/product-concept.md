# Продукт для абитуриентов: платформа осознанного выбора НПС

## 1. Описание продукта

Это цифровой продукт для абитуриентов и приёмной комиссии, который помогает сделать выбор направления обучения более понятным, осознанным и управляемым.

С одной стороны, продукт снижает тревожность абитуриента: помогает понять различия между НПС, оценить свои шансы, разобраться в сложности обучения и увидеть карьерные перспективы.

С другой стороны, продукт даёт приёмной комиссии инструменты аналитики: показывает интерес к направлениям, помогает понять, что именно путает абитуриентов, и позволяет продвигать нужные вузу программы.

---

## 2. Главная идея

Поступающий часто выбирает направление почти вслепую:

- не до конца понимает, чем отличаются программы;
- не знает, где больше программирования, где больше инженерии, где сильнее математика;
- боится ошибиться с выбором и потерять несколько лет;
- не понимает, насколько будет сложно учиться;
- не видит прозрачной связи между направлением и будущей карьерой.

Продукт должен убрать эту неопределённость и превратить выбор НПС в понятный процесс сравнения, подбора и анализа.

---

## 3. Для кого создаётся продукт

## Абитуриенты

Основная аудитория продукта — будущие студенты, которым нужно выбрать направление обучения.

### Основные потребности абитуриента

- понять различия между направлениями;
- увидеть, чему реально учат на каждом НПС;
- оценить, куда он подходит больше;
- снизить страх ошибки при выборе;
- понять сложность учёбы;
- увидеть карьерные перспективы;
- получить более персонализированную рекомендацию.

### Основные страхи абитуриента

- выбрать не то направление;
- не пройти из-за неправильной стратегии;
- не справиться с обучением;
- не понять, что ждёт после выпуска.

---

## Приёмная комиссия

Вторая ключевая аудитория — сотрудники приёмной комиссии и представители университета, которым важно понимать спрос и управлять им.

### Основные потребности приёмной комиссии

- видеть, какие направления интересуют абитуриентов;
- понимать, какие направления чаще сравнивают;
- находить точки путаницы и барьеры выбора;
- снижать нагрузку на консультации за счёт понятного интерфейса;
- продвигать нужные направления мягко и нативно;
- собирать статистику по интересу пользователей.

---

## 4. Проблема, которую решает продукт

Сейчас у абитуриента нет удобного инструмента, который бы одновременно:

- объяснял, чем отличаются НПС;
- показывал содержательные различия между программами;
- помогал подобрать подходящие варианты;
- связывал выбор направления с интересами, способностями и карьерными ожиданиями;
- делал это простым и понятным языком.

Из-за этого выбор часто строится на обрывках информации, стереотипах, советах знакомых или случайных впечатлениях.

---

## 5. Ценность продукта

## Для абитуриента

Продукт даёт:

- ясность;
- сравнимость;
- персонализацию;
- снижение тревоги;
- понимание шансов, сложности обучения и карьерных перспектив.

## Для приёмной комиссии

Продукт даёт:

- аналитику спроса;
- понимание страхов и путаницы;
- инструмент улучшения навигации по направлениям;
- возможность продвигать стратегически важные НПС;
- данные для принятия решений.

---

## 6. Состав продукта

Ниже описаны ключевые модули продукта в логичном порядке.

## 6.1. Сравнение НПС

Основной модуль продукта.

### Задача

Создать удобный инструмент сравнения НПС в понятном формате: таблица, карточки, блоки различий, акценты по ключевым параметрам.

### Что должен показывать модуль

- название направления;
- краткое позиционирование;
- ключевые различия;
- чему научат;
- акцент на программирование / математику / железо / аналитику / ИИ и т.д.;
- сложность обучения;
- возможные карьерные траектории;
- дополнительные особенности.

### Зачем он нужен

Это центральная точка принятия решения. Именно здесь пользователь должен быстро понять:

- чем отличаются направления;
- какое из них ближе ему;
- какие варианты стоит изучить дальше.

---

## 6.2. Чему научим

Модуль может быть встроен в сравнение НПС или представлен как отдельный текстовый блок на странице направления.

### Задача

Показать простым языком, какие знания, навыки и компетенции получает студент на каждом НПС.

### Что важно отразить

- какие предметные области изучаются;
- какие типы задач решает выпускник;
- какие технологии и подходы встречаются в обучении;
- на что делается основной акцент программы.

### Ценность

Абитуриенту важно видеть не только формальное название направления, но и реальное содержание обучения.

---

## 6.3. Профтест

Отдельная страница с профориентационным тестом.

### Задача

Помочь абитуриенту понять, какие НПС подходят ему больше, исходя из интересов, склонностей и предпочтений.

### Функции

- вопросы разного типа;
- итоговая интерпретация;
- рекомендации по НПС;
- переход к сравнению рекомендованных направлений.

### MVP implementation note

- первый profile test flow короткий и детерминированный;
- рекомендации объясняются через интересы, оси и предметные блоки;
- результат должен выдавать 2-3 направления, а не один "правильный" ответ;
- переход из профтеста в сравнение обязателен;
- логика рекомендаций не смешивается с продвижением направлений.

### Роль в продукте

Профтест не заменяет сравнение, а приводит пользователя к нему.
Идеальная логика:
**прошёл тест → получил рекомендации → сразу перешёл к сравнению подходящих НПС**.

---

## 6.4. Студенческая активность

Отдельный продуктовый блок про внеучебную жизнь.

### Задача

Показать, что университет — это не только учёба, но и среда, сообщество, активность, события и возможности.

### Особенность

Этот блок допускает большую творческую свободу в подаче, визуале и формате контента.

### Возможное содержание

- студенческие организации;
- мероприятия;
- проекты;
- кружки;
- атмосфера студенческой жизни;
- истории и активности.

---

## 6.5. Дашборд

Внутренний инструмент для приёмной комиссии.

### Задача

Показывать ключевые метрики посещения сайта и поведения пользователей.

### Что может включать

- посещаемость страниц;
- популярные направления;
- точки входа;
- наиболее частые сравнения;
- переходы после профтеста;
- глубину просмотра.

### Ценность

Дашборд превращает продукт из просто информационного сайта в систему принятия решений.

---

## 6.6. Сбор статистики

Механизм аналитики интереса пользователей.

### Задача

Собирать данные о популярности направлений и поведении абитуриентов.

### Что важно собирать

- какие НПС чаще открывают;
- какие НПС чаще сравнивают;
- какие рекомендации чаще получают;
- какие страницы вызывают больший интерес;
- какие направления недополучают внимание.

### Зачем это нужно

Эти данные помогают:

- улучшать сайт;
- корректировать коммуникацию;
- выявлять неочевидный спрос;
- принимать решения по продвижению.

---

## 6.7. Продвижение направлений

Функция для приёмной комиссии.

### Задача

Дать возможность выбирать направления, которые нужно продвигать сильнее.

### Как это может работать

- выбранные НПС чаще попадают в рекомендации;
- появляются в блоках “обратите внимание”;
- нативно подмешиваются в пользовательские сценарии;
- продвигаются мягко, без навязчивости.

### Почему это важно

Продукт должен не только отражать спрос, но и помогать университету управлять им.

---

## 7. Пользовательские сценарии

## Сценарий 1. Абитуриент уже примерно знает, что хочет

1. Заходит на сайт.
2. Выбирает несколько НПС.
3. Сравнивает их.
4. Читает, чему научат.
5. Делает более осознанный выбор.

## Сценарий 2. Абитуриент пока не понимает, куда идти

1. Заходит на сайт.
2. Проходит профтест.
3. Получает рекомендации.
4. Переходит к сравнению подходящих НПС.
5. Изучает различия и выбирает наиболее подходящий вариант.

## Сценарий 3. Приёмная комиссия анализирует спрос

1. Открывает дашборд.
2. Смотрит, какие направления наиболее популярны.
3. Анализирует, что чаще сравнивают.
4. Выявляет слабые направления.
5. Настраивает продвижение нужных НПС.

## 8. MVP продукта

Минимально жизнеспособная версия продукта должна решать главную задачу: помогать абитуриенту понимать различия между НПС и делать более осознанный выбор.

### Что входит в MVP

- страница сравнения НПС;
- текстовые описания “чему научим”;
- профтест с рекомендациями;
- базовый сбор статистики;
- внутренний дашборд;
- возможность продвигать отдельные направления.

### Что можно развивать позже

- более сложная персонализация;
- расширенная аналитика;
- сценарии под разные типы абитуриентов;
- интеграция с реальными данными о поступлении;
- контент про студенческую жизнь в richer-формате;
- A/B-тесты и автоматические рекомендации.

---

## 9. Принципы продукта

### Понятность

Всё должно быть объяснено простым человеческим языком, а не канцеляритом.

### Сравнимость

Пользователь должен быстро видеть различия, а не читать длинные абзацы без структуры.

### Персонализация

Продукт должен подстраивать рекомендации под интересы пользователя.

### Честность

Нужно не рекламировать направления абстрактно, а объяснять их реальные особенности и сложность.

### Польза для двух сторон

Продукт создаётся не только для абитуриента, но и для приёмной комиссии.

## 12. Статус концепции

Текущая концепция продукта опирается на две ключевые идеи:

1. помочь абитуриенту выбрать осознанно;
2. дать приёмной комиссии инструмент анализа и управления спросом.

## Именно сочетание этих двух задач делает продукт не просто справочником, а полноценной системой поддержки выбора и принятия решений.

## MVP implementation note

The current `feature/nps-comparison` iteration uses mock data first.

The mock source already follows the richer academic example shape that includes:

- program code
- qualification
- department
- study duration
- budget and paid seats
- tuition cost
- passing score history
- subjects with blocks, departments, and hours

The first applicant-facing comparison flow does not need to expose every one of these fields immediately.

The first UI iteration is expected to focus on:

- catalog visibility
- direction detail clarity
- comparison entry
- comparison of the most decision-useful fields

Real database integration for this richer structure will be connected later through repository and adapter replacement, not through a rewrite of the public pages.

## MVP implementation note

The current `feature/nps-comparison` branch implements the first applicant-facing
catalog, detail, and comparison flow on mock data derived from `hosts.txt`.

What is intentionally real already:

- the public routes
- the repository boundaries
- the comparison selection and validation logic
- analytics event emission for key applicant actions
- unit, integration, and smoke coverage for the main path

What is intentionally deferred:

- the final academic database model
- real subject, department, and curriculum normalization rules
- the final Prisma mapping for the richer dataset

The follow-up database integration should replace repository adapters, not the
public pages or comparison contracts.
